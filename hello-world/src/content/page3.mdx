## ・とはいえ、どういった観点でテストを行えばいいのか？
## ・変化が多い中で逐一テストコードも修正していったら結局爆発するのでは？

<div style={{  color: 'tomato' }}>
    <h2>→フロントにあったテストコードを書こう</h2>
</div>

従来のテストは、コンポーネント(モジュール、クラス、関数)ごとに単体テストを行ってきた。
これももちろん大事だけれども、それらを組み合わせたときに正しい状態になっていることを担保しなきゃいけない。
だから、Integreationよりのテストを書いていこう。

Bad
```js
describe("ヘッダーコンポーネントのテスト", () => {
    describe("isLoginがfalseの場合", () => {
        test.todo("<div></div>になること", () => {
            // 実際のテストコードは割愛
        })
    })
    describe("isLoginがtrueの場合", () => {
        test.todo("<div><div class="user-info">ユーザー名：おにぎり</div></div>になること")
    })
})
```

Good
```js
describe("トップページのテスト", () => {
    describe("ユーザーが未ログインの場合", () => {
        test.todo("ヘッダーに、ユーザー情報アイコンが表示されないこと")
    })
    describe("ユーザーがログイン済みの場合", () => {
        test.todo("ヘッダーに、ユーザー情報アイコンが表示されていること")
    })
})
```
